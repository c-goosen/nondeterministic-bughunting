# nondeterministic-bughunting

Using CLI agentic tools to bughunt. Skills, code, etc.

## Security audit toolkit

This repo bundles an end-to-end, agentic security-audit pipeline under
[`.claude/skills/`](.claude/skills/). A single orchestrator skill,
[`bughunt-security-audit`](.claude/skills/bughunt-security-audit/SKILL.md), sequences pipeline
skills across **seven phases** and applies a manual-audit methodology at each
step:

```
bughunt-threat-model  →  bughunt-vuln-scan  →  bughunt-triage  →  bughunt-patch  →  bughunt-exploit-payloads  →  bughunt-audit-report
   (recon)        (hunt)      (validate)  (fix)      (PoC inputs)         (index.html)
```

| Phase | Methodology (Cloudflare) | Pipeline skill | Primary artifact |
|---|---|---|---|
| 1. Recon | [`RECONNAISSANCE.md`](.claude/skills/bughunt-security-audit/RECONNAISSANCE.md) | [`bughunt-threat-model`](.claude/skills/bughunt-threat-model/SKILL.md) | `THREAT_MODEL.md`, `architecture.md` |
| 2. Hunt | [`HUNTING.md`](.claude/skills/bughunt-security-audit/HUNTING.md), [`ATTACK-CLASSES.md`](.claude/skills/bughunt-security-audit/ATTACK-CLASSES.md) | [`bughunt-vuln-scan`](.claude/skills/bughunt-vuln-scan/SKILL.md) | `VULN-FINDINGS.json` |
| 3. Validate | [`VALIDATION-AND-REPORTING.md`](.claude/skills/bughunt-security-audit/VALIDATION-AND-REPORTING.md) §Phase 3 | [`bughunt-triage`](.claude/skills/bughunt-triage/SKILL.md) | `TRIAGE.json` |
| 3.7 Execute-verify *(optional)* | Cloudflare VDH (PoC-as-test, sandboxed) | [`bughunt-verify`](.claude/skills/bughunt-verify/SKILL.md) | `VERIFY.json`, `VERIFY/` |

Run it by invoking the `bughunt-security-audit` skill against a target codebase. **The
run is not complete until Phase 7 writes `index.html`** — a self-contained dark
report generated by
[`bughunt-audit-report/generate-report.py`](.claude/skills/bughunt-audit-report/generate-report.py)
(layout reference: `~/security-audit-skill/openclaw/run-1/index.html`).

**Triage verification:** Phase 3 (`bughunt-triage`) runs each verifier vote on a
different model and sends split outcomes to a judge on the largest capable
model available (see [`bughunt-triage/SKILL.md`](.claude/skills/bughunt-triage/SKILL.md) Phase 0e).

### Vulnerability-harness upgrades (Cloudflare methodology)

Beyond the vendored Cloudflare `security-audit` docs, the pipeline folds in
techniques from Cloudflare's
[_Build your own vulnerability harness_](https://blog.cloudflare.com/build-your-own-vulnerability-harness/):

- **Per-finding threat gate** — every hunt finding must state `attacker →
  boundary`; triage exclusion rule 17 drops "vacuous" findings where the
  attacker already holds the capability the bug supposedly grants.
- **Disprove-only validation** — `bughunt-vuln-scan`'s Step 3b is an adversarial
  *disprove* pass (records the strongest counter-argument, never drops), and
  triage verifiers/judge are explicitly disprove-only: they may reject or
  confirm, never invent new findings or broaden scope.
- **Coverage gapfill** — `bughunt-vuln-scan` Step 2b runs a bounded second hunt
  over files no focus area covered, so an unscoped file isn't an unread file.
- **Feedback loop** — the orchestrator's Phase 3.5 mines triage rejects into a
  `feedback.md` (FP-avoidance rules + missed-area leads) that seeds the next
  run's `--fp-rules` / `--extra` and threat model.
- **Execution verification** — the optional
  [`bughunt-verify`](.claude/skills/bughunt-verify/SKILL.md) skill writes a
  proof-of-concept **as a test against the untouched codebase**, runs it in a
  sandbox (network-denied, target read-only, via
  [`run-sandboxed.sh`](.claude/skills/bughunt-verify/run-sandboxed.sh) using
  `bwrap`/`unshare`/`firejail`/`docker`), and promotes a finding to
  `execution_verified` only on the observed signal — not the exit code. With
  `--with-patch` it re-runs the PoC against a patched copy to confirm the fix
  closes it. **This is the one skill that runs target code — authorized,
  sandboxed targets only.**

### Deterministic pre-scan

The `bughunt-vuln-scan` stage front-loads five external static scanners before the
LLM fan-out, so the agents start from real, reproducible signal:

- **[semgrep](https://github.com/semgrep/semgrep)** — pattern-based SAST.
  Step 0 uses `bughunt-vuln-scan/run-semgrep.sh` (bundled `semgrep/rules.yaml`, scans
  `targets/<name>` or a repo path locally — no `--config auto` registry fetch).
- **[osv-scanner](https://github.com/google/osv-scanner)** — known-vulnerable
  dependencies via [OSV.dev](https://osv.dev).
- **[grype](https://github.com/anchore/grype)** — image/filesystem CVEs, run
  only when a Dockerfile is present.
- **[gitleaks](https://github.com/gitleaks/gitleaks)** — secrets in code and
  git history (always-on).
- **[checkov](https://github.com/bridgecrewio/checkov)** — infrastructure-as-code
  misconfig, run only when IaC (Terraform, k8s, CloudFormation, …) is present.

Install them once with:

```
bash .claude/skills/bughunt-vuln-scan/setup-tools.sh
```

### Historical context (Security Context)

For targets that are public GitHub repos, `bughunt-vuln-scan` and `bughunt-triage` also
pull cached context from [Security Context](https://securitycontext.dev)
(`securitycontext.dev`) — a free, unauthenticated API/MCP service that
mines a repo's real fix-commit history and disclosed CVEs into recurring
weak spots and a ranked variant-lead backlog. It's a strong prior for
"where the next bug is most likely to surface" and is treated as advisory
context (leads to confirm/refute), never an auto-promoted finding. This is
best-effort and network-optional: non-GitHub targets, or an unreachable
service, just skip it. See [`securitycontext.dev/docs`](https://securitycontext.dev/docs)
for the full API/MCP surface.

### Other agentic CLIs

`.opencode/skills` and `.cursor/skills` are symlinks to `.claude/skills`, so
the same skill set is available to OpenCode and Cursor as well.

Re-running the pipeline through the Cursor CLI agent with a non-Anthropic
model — e.g. **GLM 5.2** or **Kimi K2.7** — is recommended as a second pass:
the same skills prompted through a different model surface different
findings, which prevents single-model blind spots (and avoids burning
Claude usage on repeat scans). From the repo root:

```
cursor-agent --model glm-5.2  "Run the bughunt-security-audit skill against targets/<repo>"
cursor-agent --model kimi-k2.7 "Run the bughunt-security-audit skill against targets/<repo>"
```

(Model IDs follow whatever Cursor exposes in its `/model` picker — adjust
to the exact slug your Cursor version lists.) Diff the resulting
`VULN-FINDINGS.json` files, then feed the union into a single `/bughunt-triage`
pass to dedupe and validate across models.

### Harness config

[`.claude/settings.json`](.claude/settings.json) pins the model and
pre-approves the read-only shell commands and scanner invocations the
pipeline needs (semgrep, osv-scanner, grype, gitleaks, checkov, `git`,
`gh api`, …), so an audit run doesn't stall on permission prompts.
